Codex Plugin Virtual BA Supervisor 需求澄清 验收驱动开发 BA / Coder 辩论
产品负责人
Phase 0 MVP

VB-S 全称 Virtual BA Supervisor，是一款将现代 IT 技术项目经理带入 IDE 的 Codex 插件。它不替代编码 Agent，而是监督 AI 编码过程：将模糊需求澄清为可执行任务，持续汇总进度、风险与决策，在内部组织 BA 和 Coder 讨论，并只把真正影响产品方向、成本、性能、合规或跨团队范围的决定升级给用户。最终通过验收证据与修复提示词形成从需求到交付的闭环。

需求澄清与任务三件套

/clarify [requirement] mode=light|standard|deep 将模糊描述转化为结构化需求，并一次生成三项可直接交付给 Codex 的内容：可执行提示词、任务拆解清单和验收标准草案。不同深度模式让用户可以在快速补全与完整需求分析之间选择。

三级监督模型：减少无效打扰

L1 自动处理 优先利用本地代码、文档、历史决策和默认值解决问题；L2 内部讨论 组织 BA 与 Coder 在插件内部收敛可执行方案；只有涉及产品方向、成本、性能、合规、跨团队范围，或三轮讨论后仍无法解决的问题，才进入 L3 用户决策。

进度、风险与决策记忆

/vbs report 汇总总体进度、已完成和进行中任务、阻塞项、测试状态与后续计划；/vbs risks 扫描未解决假设、阻塞和决策债务；/vbs decisions 保留历史决定及其理由。报告频率与内容可通过 /vbs config 配置。

DeepDive 证据化多轮辩论

/deepdive [requirement] X=[rounds] 启动结构化多轮讨论。BA 与 Coder 分别给出观点、证据、反方意见、风险和决策影响，每个关键论点必须引用项目证据或官方资料；最终结果可让用户直接选择“采纳 A”“采纳 B”或“继续追问”。

验收驱动与自动修复闭环

每个已完成功能都必须关联验收测试脚本或命令，覆盖成功路径与失败/边界路径，并附带真实执行状态、证据和剩余缺口。VB-S 不允许在没有执行证据时将功能标记为通过；验收失败后会自动生成 Repair Prompt，进入下一轮 Codex 修复。

Codex Runtime 与定时监督

Runtime 适配层将“发送给 Codex”路由到任务创建或消息发送工具，将 Stop Hook 完成信号转换为验收跟进，并在失败时衔接修复提示词。报告配置还可以生成 Codex Automation，例如工作日 18:00 自动输出包含验收覆盖率的监督报告。
